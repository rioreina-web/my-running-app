# Migration Ledger Reconciliation — 2026-06-11

**Why:** Repo and prod migration histories have diverged. Until `supabase migration list` shows them in agreement, `supabase db push` cannot be trusted, and the automated deploy pipeline (see docs/ops-delivery-roadmap-2026-06-10.md, Phase 1) cannot be built on it. This doc is the exact reconciliation plan, generated by diffing the repo's `supabase/migrations/` against prod's `supabase_migrations.schema_migrations` (project `aqdijapxmjqaetursrde`).

**State at time of audit:** repo = 100 files, prod = 96 ledger entries, 77 match on name+version.

## Strategy

Direction of repair: **change the repo to match prod, never the reverse.** Prod's ledger reflects what actually ran. Renaming repo files to prod's versions requires no prod writes and no `migration repair`. After the renames + the recovered file (already added), the only remaining deltas are 5 genuinely-unapplied migrations, each of which needs a conscious apply/fold/drop decision.

## Hard rules going forward

1. No ad-hoc migrations via dashboard/MCP `apply_migration` against prod. Every migration is a repo file applied via `db push`. (The 17 re-stamps and 2 ghost entries all came from ad-hoc applies.)
2. Migration filenames: strictly `YYYYMMDDHHMMSS_name.sql` — the `20260128_152000_user_profile.sql` malformed-timestamp collision cost 5 months of user_profiles workarounds.
3. Migrations are append-only (CLAUDE.md rule #5); rollback = compensating migration.
