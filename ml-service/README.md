## Why it's parked

This FastAPI service was built early as the home for ML-driven race predictions and injury risk scoring. The architecture is solid — JWT auth, ownership checks, CORS via env, Sentry, non-root Docker — but **nothing in iOS or web actually calls it**.

The iOS app uses a heuristic local prediction in `FitnessPredictorService.swift` instead. That works well for V1 and avoids:

- Network latency on every prediction
- Railway hosting costs
- Operational complexity (env vars, deploys, monitoring)
- Cold-start XGBoost model with no real data to train on

## What it would do (if deployed)

Three endpoints (all JWT-authed, ownership enforced):

- `POST /predict-fitness` — XGBoost race time prediction (mile/5K/10K/half/marathon)
- `POST /injury-risk` — ACWR + monotony-based risk score
- `POST /training-summary` — quick weekly stats rollup

## When to revisit

Deploy when **all three** are true:

1. **30+ active users** with **60+ days of workouts each** in `workout_features` table — minimum to train a non-trivial XGBoost model
2. A real product reason — e.g., personalized injury prevention recommendations or coach dashboards needing aggregate risk scores
3. Willingness to add ~$15-25/mo Railway hosting + the ops burden

Until then: keep the code, don't pay the hosting bill.

## To deploy later

Railway deployment config (`railway.toml`) was intentionally removed on 2026-04-16
to prevent accidental redeployment while the service is parked. If the Railway project
is still live, **delete it from the Railway dashboard** — there's no reason to keep an
unauthenticated endpoint running.

When ready to deploy, recreate `railway.toml`:

```toml
[build]
builder = "dockerfile"
dockerfilePath = "Dockerfile"

[deploy]
startCommand = "uvicorn app.main:app --host 0.0.0.0 --port $PORT"
healthcheckPath = "/health"
healthcheckTimeout = 10
restartPolicyType = "on_failure"
restartPolicyMaxRetries = 3
```

Then:

```bash
# 1. Install + auth
brew install railway && railway login

# 2. Init from this dir
cd ml-service
railway init

# 3. Set env vars (use Vault/1Password — never commit these)
railway variables set \
  SUPABASE_URL=<your-supabase-url> \
  SUPABASE_SERVICE_ROLE_KEY=<from-vault> \
  SUPABASE_JWT_SECRET=<from-supabase-dashboard> \
  ALLOWED_ORIGINS=https://your-prod-domain.com \
  SENTRY_DSN=https://...@sentry.io/...

# 4. Deploy
railway up

# 5. Wire iOS + web to call it
#    - Set ML_SERVICE_URL in Vercel env vars
#    - Replace FitnessPredictorService.swift's local logic with a network call
```

## What's already production-ready in this code

When you do deploy, no rework needed:

- ✅ JWT validation middleware (`app/auth.py`)
- ✅ Ownership enforcement (token `sub` must match `user_id` in body)
- ✅ Bounded input params (`days: int = Field(ge=1, le=730)`)
- ✅ CORS via `ALLOWED_ORIGINS` env var
- ✅ Env var validation on startup (fails fast)
- ✅ Non-root Docker user (`appuser`)
- ✅ Sentry FastAPI integration (toggled via `SENTRY_DSN`)
- ✅ Pinned dependencies in `requirements.txt`
